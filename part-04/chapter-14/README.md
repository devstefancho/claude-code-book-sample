# Chapter 14. 스킬 크리에이터로 스킬 평가 및 고도화하기

> 프롬프트: [prompts.md](./prompts.md) | 코드: [code.md](./code.md)

앞서 만든 `component-design-reviewer` 스킬로 리액트 컴포넌트를 성공적으로 리뷰했습니다. 하지만 스킬을 계속 사용하다 보면 아쉬운 점이나 부작용이 발견되기 마련입니다. 이때 수동으로 마구잡이식 수정을 하면 잘 동작하던 기능마저 망가질 위험이 있습니다. Anthropic이 제공하는 `skill-creator` 플러그인을 활용하면 기존 스킬과 새로운 스킬을 블라인드 테스트하고 정량적으로 평가하여 안전하게 스킬을 고도화할 수 있습니다.

## 평가 방식의 스킬 개선 흐름 이해하기

`/skill-creator`로 스킬 개선을 요청하면 다음 단계를 거칩니다.

1. 워크스페이스 디렉터리와 `워크스페이스/iteration` 디렉터리를 생성합니다.
2. 여러 프롬프트로 기존 스킬과 개선 스킬을 실행합니다.
3. 같은 프롬프트의 결과를 비교합니다.
4. 지정한 기준에 따라 PASS/FAIL이 결정됩니다.

평가는 병렬로 실행할 수도, 순차로 실행할 수도 있습니다. 심사위원이 3명이면 세 케이스를 한 번에 심사하고, 한 명이면 순서대로 진행하는 것과 같습니다. 모든 통과 기준을 만족했더라도 더 개선하고 싶은 점이 있으면 피드백을 주어 같은 과정을 한 번 더 반복할 수 있습니다. 첫 사이클은 `iteration-1`, 추가 평가는 `iteration-2`에서 진행되며, 같은 스킬의 개선 과정은 몇 번을 반복하든 하나의 워크스페이스에서 처리됩니다. 이렇게 하면 느낌이 아닌 정확한 평가 기준으로 스킬을 개선하게 됩니다.

## 워크스페이스의 최종 결과물 먼저 보기

스킬 개선 과정에서 생성되는 파일은 모두 `component-design-reviewer-workspace` 아래에 만들어집니다. `feedback.json`을 제외한 나머지는 클로드 코드가 자동으로 생성합니다.

- **워크스페이스**: 개선을 시작하면 가장 먼저 생성되는 격리된 환경. 이후 모든 파일이 이 아래에 들어갑니다.
- **skill-snapshot/**: 기존 스킬의 복사본.
- **improved-skill/**: 개선된 스킬 버전.
- **반복 공간(iteration-N)**: 한 번 개선할 때마다 묶이는 디렉터리. 최초 개선은 `iteration-1`에서 일어나고, 그 아래에 여러 테스트 케이스(Eval)가 이어집니다. 각 Eval 안에는 `old_skill`(기존 스킬)과 `with_skill`(개선 스킬)이 있어 하나의 프롬프트에 대한 두 결과가 기록됩니다.
- **benchmark.json**: 전체 점수가 기록되는 최종 요약.
- **feedback.json**: 더 바꾸고 싶은 점을 적어 클로드 코드에게 다시 주면 스킬을 한 번 더 개선합니다. 추가 피드백을 하지 않으면 생성되지 않습니다.
- **review.html**: 평가 결과를 쉽게 비교하고 피드백을 작성하는 웹페이지.
- **test-fixtures/**: 스킬을 테스트할 대상 샘플 코드.

워크스페이스만 잘 이해하면 평가 방식의 스킬 개선 프로세스를 완전히 이해할 수 있습니다.

자세한 디렉터리 트리는 [code.md](./code.md)를 참고하세요.

## 스킬 개선 실습하기

1. 임시 디렉터리를 만들고 챕터 13에서 만든 스킬을 다운로드합니다. (명령어는 [code.md](./code.md) 참고)
2. 클로드 코드를 켜고 `/skills` 커맨드로 `component-design-reviewer`가 설치됐는지 확인합니다.
3. `/skill-creator`를 실행하면 새 스킬 만들기 / 기존 스킬 개선 / 설명 최적화 중 무엇을 할지 안내합니다.
4. 기존 스킬 개선을 선택하고, 구체적인 개선 방향을 프롬프트로 입력합니다. 이때 **정식 Eval 루프 사용**과 **순차 실행**을 함께 요청하는 것이 중요합니다. (프롬프트는 [prompts.md](./prompts.md) 참고)
5. 클로드 코드가 테스트 케이스를 직접 만들어 줍니다. 예시에서는 Eval 0(AdminDashboard), Eval 1(CheckoutFlow), Eval 2(ProjectBoard) 세 가지 평가가 순차적으로 실행됩니다.
6. 평가 루프가 끝나면 결과 웹페이지(`review.html`)가 자동으로 생성됩니다.
7. 웹페이지에서 기존 스킬과 개선 스킬의 리뷰 결과를 비교합니다. 예시에서 기존 스킬은 4개 중 2개가 실패했지만, 개선 스킬은 4개를 모두 통과했습니다. 아쉬운 점은 피드백 입력창에 작성하고 리뷰를 제출하면 `iteration-1`에 `feedback.json`이 저장됩니다.
8. 저장한 피드백을 기반으로 한 번 더 개선을 요청하면 `iteration-2`가 생성되며 개선이 반복됩니다.

다양한 프롬프트로 비교 테스트하고 피드백으로 여러 차례 개선하면 완성도 높은 코드 리뷰 스킬을 만들 수 있습니다.

## 바이브 UP! 3초 꿀팁 — 스킬 개선 프롬프트 작성 공식

- **정확한 증상과 제약 조건**: "Props 전달이 3단계를 넘지 않아야 해"처럼 타협할 수 없는 조건을 명시합니다.
- **평가 루프 강제 호출**: "그냥 수정하지 말고 정식 Eval 루프를 사용해 줘"라고 해야 전후 비교 벤치마크 워크스페이스가 정상 작동합니다.
- **순차 실행 지시**: 대규모 작업은 "한 번에 돌리지 말고 순차적으로 실행해 줘"라고 해야 병목으로 클로드 코드가 멈추는 것을 방지합니다.
