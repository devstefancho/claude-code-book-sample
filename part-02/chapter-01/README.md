# Chapter 01. 칸반보드 프론트엔드 개발하기

> 프롬프트: [prompts.md](./prompts.md) | 코드: [code.md](./code.md)

## 어떤 순서로 만들지 생각해보기

칸반보드를 만들 때 단순히 '칸반보드 만들어'라고 요청하면 원하는 결과가 나오지 않습니다. 지시 사항과 요구 사항이 명확해야 하고, 작업의 단계가 있어야 합니다.

SDD(Spec-Driven Development)는 명세를 기반으로 개발하는 방법론입니다. 요구 사항과 단계를 합친 것으로, AI에게 잘 이야기할 수 있는 방법 중 하나입니다.

## SDD는 문서 기반의 개발

요구 사항을 먼저 작성하고 그것을 기반으로 개발합니다. 요구 사항은 프로그래밍 언어가 아니라 한글과 영어로 작성합니다.

요구 사항을 한 번에 잘 작성하는 것은 거의 불가능합니다. 보통 요구 사항에는 다음을 고려합니다:
1. 필수 기능에 대한 설명
2. 기능 구현에 필요한 기술 목록
3. 테스트 하는 방법

클로드 코드에서는 재사용 가능한 프롬프트를 **커스텀 슬래시 커맨드**라고 부릅니다.

## 이 책에서 사용할 커스텀 커맨드

| 단계 | 커맨드 | 목적 |
|------|--------|------|
| 스펙(Spec) | `/sdd:spec` | 무엇을 만들고 싶은가 |
| 플랜(Plan) | `/sdd:plan` | 무슨 기술을 사용하는가 |
| 태스크(Tasks) | `/sdd:tasks` | 실제로 어떤 작업을 하는가 |
| 임플리먼트(Implement) | `/sdd:implement` | 코드로 구현하는 단계 |

**핵심**: "무엇을 만들지 결정하고, 어떤 기술을 사용할지 선택하고, 상세한 작업 계획서를 작성하고, 이를 토대로 실제 코드로 구현한다."

## 클로드 코드 플랜모드로 실행하기

- `[Shift + Tab]`으로 모드 전환 가능
- **plan mode**: 계획을 세울 때 사용, 파일 수정 불가
- **accept edits**: 파일 수정을 허용하는 모드

플랜 모드는 아이디어 논의 단계에서 실수로 인한 구현이나 문서 생성을 방지할 수 있습니다.

## Git 프로젝트 생성하기

클로드 코드를 사용해서 프로젝트 공간을 세팅합니다. 디렉터리를 만들거나 .gitignore를 수정하는 간단한 작업도 클로드 코드를 통해서 작업합니다.

하나의 작업을 마치면 클로드 코드를 종료하고 새 세션으로 켜는 것을 습관화합니다. 한 세션이 하나의 작업만 하는 것을 원칙으로 합니다.

## 네임스페이스로 커맨드 정리하기

커스텀 커맨드의 이름은 폴더 구조를 따라갑니다:
- `.claude/commands/spec.md` → `/spec`
- `.claude/commands/sdd/spec.md` → `/sdd:spec`

네임스페이스를 사용하는 이유:
1. 클로드 코드 기본 커맨드와 이름 충돌 방지
2. SDD 방법론 커맨드들을 한 곳에 모아서 관리

## 스펙 단계: 무엇을 만들지 결정하기

스펙 단계에서는 무엇을 만들 것인지만 담습니다. 기술 요구 사항은 제외합니다.

### 생성되는 파일
1. `.claude/commands/sdd/spec.md` - 커스텀 커맨드 파일
2. `docs/frontend-setup/spec.md` - 스펙 문서 (결과물)

### 커맨드 구조 (4가지 요소)
- **Argument**: 커맨드 뒤에 오는 인자
- **Instructions**: 생성될 문서의 내용과 구조
- **Report**: 실행 후 결과 요약 및 다음 단계 안내
- **Quality**: 지켜야 할 품질 수준

MVP(Minimum Viable Product) 관점에서 서비스의 최소 기능만 담아주는 게 좋습니다. 초기 제품을 만들 때는 3~4개 정도의 요구 사항이 적당합니다.

## 플랜 단계: 기술 선택하기

스펙 단계에서 무엇을 만들지 정했다면, 이제 사용할 기술을 선택합니다. 기술 요구 사항은 기능을 따르므로, 스펙 정의가 선행되어야 합니다.

### 생성되는 파일
1. `.claude/commands/sdd/plan.md` - 커스텀 커맨드 파일
2. `docs/frontend-setup/plan.md` - 플랜 문서 (결과물)

### 플랜 문서 포함 내용
- Technology Stack (기술, 프레임워크, 라이브러리)
- Architecture Overview (고수준 시스템 설계)
- Data Models (DB 스키마, 데이터 구조)
- Component Structure (컴포넌트 계층)
- State Management (데이터 흐름)
- Security Considerations (인증, 인가, 데이터 보호)

## 태스크 단계: 작업 계획서 만들기

스펙과 플랜을 토대로 실제로 필요한 작업들을 구체적으로 나열합니다. 클로드 코드가 요소들을 조합하고 분해하여 유의미한 단위의 작업으로 만들어 줍니다.

### 생성되는 파일
1. `.claude/commands/sdd/tasks.md` - 커스텀 커맨드 파일
2. `docs/frontend-setup/tasks.md` - 태스크 문서 (결과물)

### 태스크 문서 특징
- T-001, T-002 형식의 넘버링
- Task Groups로 그룹화
- 의존성 고려하여 순서 정렬

### 몇 개의 태스크가 적당할까?

초기 프로젝트 세팅의 경우 **30~40개** 수준이 적당합니다. 50개가 넘어가면 선택 작업이 섞여 있다고 판단하고 개수를 줄이는 것이 좋습니다.

### 태스크 조정 예시
- PoC가 목적이라서 테스트 코드가 필요 없는 경우
- 특정 작업의 우선순위를 바꾸고 싶은 경우
- 선택 기능 제외하고 필수 기능 위주로 작업하는 경우
- 스타일 작업을 최소화하고 싶은 경우

## 임플먼트 단계: 코드 구현하기

태스크에 작성한 작업을 하나씩 코드로 구현합니다. 앞선 단계에서 만든 3개의 마크다운 파일(spec.md, plan.md, tasks.md)을 기반으로 실제 코드를 생성합니다.

### 생성되는 파일
1. `.claude/commands/sdd/implement.md` - 커스텀 커맨드 파일
2. 실제 소스 코드 파일들 (결과물)

### 구현 전략
- `--all` 플래그: 모든 작업 일괄 구현
- 플래그 없음: 그룹별로 구현 후 사용자 승인 대기

단계적으로 접근하면 속도가 느리게 느껴질 수 있지만, 구현의 성공 확률이 급격히 증가합니다.
