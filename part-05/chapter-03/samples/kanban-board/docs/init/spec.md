# Kanban Board 웹 서비스 명세서

## Original Requirements

kanban-app 칸반 보드 웹 서비스를 만들려고 한다. 상태는 할 일, 진행 중, 완료 이렇게 3가지가 있으며, Drag & Drop 방식으로 카드를 옮기고 옮긴 위치에 따라 상태가 변경되어야 한다.

## Overview

칸반 보드 웹 서비스는 사용자가 작업(Task)을 시각적으로 관리할 수 있는 웹 기반 애플리케이션입니다. 사용자는 카드 형태의 작업을 생성하고, 드래그 앤 드롭 인터랙션을 통해 "할 일", "진행 중", "완료" 3가지 상태 간에 자유롭게 이동시킬 수 있습니다.

**무엇을 만드는지 (What):**

- 3개의 칼럼(할 일, 진행 중, 완료)으로 구성된 칸반 보드
- 드래그 앤 드롭 기능을 지원하는 작업 카드
- 직관적이고 반응성 높은 웹 인터페이스

**왜 필요한지 (Why):**

- 작업의 진행 상태를 시각적으로 파악
- 드래그 앤 드롭을 통한 직관적인 작업 관리
- 개인 또는 팀의 생산성 향상

**누가 사용하는지 (Who):**

- 개인 작업을 관리하는 사용자
- 프로젝트 진행 상황을 추적하려는 사용자
- 시각적 작업 관리 도구가 필요한 모든 사용자

## Goals

1. **직관적인 작업 관리 시스템 구축**
   - 사용자가 5분 이내에 첫 카드를 생성하고 이동할 수 있어야 함
   - 별도의 튜토리얼 없이 드래그 앤 드롭 기능을 즉시 사용 가능

2. **원활한 드래그 앤 드롭 경험 제공**
   - 드래그 시작부터 드롭까지 60fps 이상의 부드러운 애니메이션
   - 카드 이동 시 즉각적인 시각적 피드백 제공

3. **명확한 작업 상태 시각화**
   - 3개 칼럼(할 일, 진행 중, 완료)을 통한 명확한 작업 상태 구분
   - 각 칼럼별 카드 개수를 한눈에 파악 가능

4. **안정적인 상태 관리**
   - 카드 이동 시 데이터 손실 없이 상태 변경 보장
   - 페이지 새로고침 후에도 작업 상태 유지

## User Stories

### US-001: 카드 생성

```
As a 사용자,
I want to 새로운 작업 카드를 생성하고,
So that 해야 할 일을 칸반 보드에 추가할 수 있다.
```

**Acceptance Criteria:**

- [ ] "할 일" 칼럼에서 새 카드 추가 버튼을 클릭할 수 있어야 함
- [ ] 카드 제목을 입력할 수 있어야 함
- [ ] 카드 설명(선택 사항)을 추가할 수 있어야 함
- [ ] 생성된 카드가 "할 일" 칼럼에 표시되어야 함

### US-002: 카드 드래그 앤 드롭으로 상태 변경

```
As a 사용자,
I want to 카드를 드래그하여 다른 칼럼으로 이동하고,
So that 작업의 진행 상태를 쉽게 업데이트할 수 있다.
```

**Acceptance Criteria:**

- [ ] 카드를 마우스로 드래그할 수 있어야 함
- [ ] 드래그 중인 카드가 시각적으로 구분되어야 함
- [ ] 드롭 가능한 영역이 명확하게 표시되어야 함
- [ ] 카드를 다른 칼럼에 드롭하면 상태가 자동으로 변경되어야 함
- [ ] 같은 칼럼 내에서 카드 순서를 변경할 수 있어야 함

### US-003: 카드 조회

```
As a 사용자,
I want to 각 칼럼의 카드를 확인하고,
So that 현재 진행 중인 작업 상황을 파악할 수 있다.
```

**Acceptance Criteria:**

- [ ] 각 칼럼에 속한 모든 카드가 표시되어야 함
- [ ] 카드에 제목과 간략한 정보가 표시되어야 함
- [ ] 각 칼럼의 카드 개수가 표시되어야 함
- [ ] 카드를 클릭하여 상세 정보를 볼 수 있어야 함

### US-004: 카드 수정

```
As a 사용자,
I want to 기존 카드의 내용을 수정하고,
So that 작업 정보를 최신 상태로 유지할 수 있다.
```

**Acceptance Criteria:**

- [ ] 카드를 클릭하여 편집 모드로 전환할 수 있어야 함
- [ ] 카드 제목을 수정할 수 있어야 함
- [ ] 카드 설명을 수정할 수 있어야 함
- [ ] 수정 사항이 즉시 반영되어야 함
- [ ] 취소 옵션이 제공되어야 함

### US-005: 카드 삭제

```
As a 사용자,
I want to 불필요한 카드를 삭제하고,
So that 보드를 깔끔하게 유지할 수 있다.
```

**Acceptance Criteria:**

- [ ] 카드에서 삭제 버튼을 찾을 수 있어야 함
- [ ] 삭제 전 확인 메시지가 표시되어야 함
- [ ] 삭제 후 카드가 즉시 제거되어야 함
- [ ] 실수로 삭제한 경우 복구할 수 있는 방법이 있어야 함 (선택 사항)

### US-006: 작업 진행 상황 파악

```
As a 사용자,
I want to 각 단계별 작업량을 한눈에 보고,
So that 전체적인 작업 진행 상황을 파악할 수 있다.
```

**Acceptance Criteria:**

- [ ] 각 칼럼의 헤더에 카드 개수가 표시되어야 함
- [ ] 전체 카드 수 대비 완료된 카드 비율을 확인할 수 있어야 함
- [ ] 빈 칼럼도 명확하게 식별 가능해야 함

### US-007: 반응형 인터페이스 사용

```
As a 모바일/태블릿 사용자,
I want to 다양한 화면 크기에서 칸반 보드를 사용하고,
So that 어떤 디바이스에서든 작업을 관리할 수 있다.
```

**Acceptance Criteria:**

- [ ] 모바일에서 칼럼이 세로로 스크롤 가능해야 함
- [ ] 터치 디바이스에서 드래그 앤 드롭이 작동해야 함
- [ ] 태블릿에서 최소 2개 칼럼이 동시에 보여야 함
- [ ] 데스크톱에서 3개 칼럼이 모두 보여야 함

## Functional Requirements

### 카드 관리

**FR-001: 카드 생성** (Must-have)

- **Description:** 사용자가 새로운 작업 카드를 생성할 수 있어야 함
- **Acceptance Criteria:**
  - [ ] 각 칼럼에 "카드 추가" 버튼이 있어야 함
  - [ ] 제목 필드는 필수 입력 항목이어야 함 (최대 200자)
  - [ ] 설명 필드는 선택 항목이어야 함 (최대 2000자)
  - [ ] 생성 시 자동으로 현재 날짜/시간이 기록되어야 함
  - [ ] 빈 제목으로는 카드를 생성할 수 없어야 함

**FR-002: 카드 조회** (Must-have)

- **Description:** 사용자가 모든 카드를 상태별로 조회할 수 있어야 함
- **Acceptance Criteria:**
  - [ ] 각 칼럼에 해당 상태의 카드만 표시되어야 함
  - [ ] 카드가 생성 날짜 역순으로 정렬되어야 함
  - [ ] 카드 클릭 시 전체 내용을 볼 수 있어야 함
  - [ ] 각 칼럼 헤더에 카드 개수가 표시되어야 함

**FR-003: 카드 수정** (Must-have)

- **Description:** 사용자가 기존 카드의 내용을 수정할 수 있어야 함
- **Acceptance Criteria:**
  - [ ] 카드를 더블클릭하거나 편집 버튼을 클릭하여 수정 모드 진입
  - [ ] 제목과 설명을 수정할 수 있어야 함
  - [ ] 수정 시 마지막 수정 날짜/시간이 업데이트되어야 함
  - [ ] 저장/취소 버튼이 제공되어야 함
  - [ ] ESC 키로 수정을 취소할 수 있어야 함

**FR-004: 카드 삭제** (Must-have)

- **Description:** 사용자가 불필요한 카드를 삭제할 수 있어야 함
- **Acceptance Criteria:**
  - [ ] 각 카드에 삭제 버튼이 있어야 함
  - [ ] 삭제 전 확인 대화상자가 표시되어야 함
  - [ ] 삭제 후 해당 칼럼의 카드 개수가 자동 업데이트되어야 함
  - [ ] 삭제 작업은 되돌릴 수 없음을 사용자에게 알려야 함

### 드래그 앤 드롭

**FR-005: 카드 드래그 시작** (Must-have)

- **Description:** 사용자가 카드를 드래그할 수 있어야 함
- **Acceptance Criteria:**
  - [ ] 카드를 마우스로 클릭하고 유지하면 드래그 모드 활성화
  - [ ] 드래그 중인 카드가 시각적으로 구분되어야 함 (투명도/그림자)
  - [ ] 원래 위치에 placeholder가 표시되어야 함
  - [ ] 드래그 중 커서가 적절하게 변경되어야 함

**FR-006: 드롭 영역 표시** (Must-have)

- **Description:** 드래그 중 드롭 가능한 영역이 명확하게 표시되어야 함
- **Acceptance Criteria:**
  - [ ] 드래그 중 다른 칼럼에 호버 시 드롭 가능 영역이 하이라이트되어야 함
  - [ ] 카드 사이의 삽입 위치가 시각적으로 표시되어야 함
  - [ ] 드롭 불가능한 영역에서는 금지 커서가 표시되어야 함

**FR-007: 카드 드롭 및 상태 변경** (Must-have)

- **Description:** 카드를 드롭하면 새로운 위치와 상태가 즉시 반영되어야 함
- **Acceptance Criteria:**
  - [ ] 다른 칼럼에 드롭 시 카드 상태가 자동 변경되어야 함
  - [ ] 같은 칼럼 내에서 드롭 시 순서만 변경되어야 함
  - [ ] 드롭 후 부드러운 애니메이션으로 카드가 이동해야 함
  - [ ] 원래 칼럼과 새 칼럼의 카드 개수가 자동 업데이트되어야 함
  - [ ] 드래그 취소(ESC 키) 시 카드가 원래 위치로 돌아가야 함

**FR-008: 터치 디바이스 지원** (Should-have)

- **Description:** 모바일/태블릿에서 터치로 드래그 앤 드롭이 가능해야 함
- **Acceptance Criteria:**
  - [ ] 터치 앤 홀드로 드래그 모드 활성화
  - [ ] 터치로 카드를 다른 칼럼으로 이동 가능
  - [ ] 터치 제스처가 스크롤과 충돌하지 않아야 함
  - [ ] 롱 프레스 후 햅틱 피드백 제공 (지원 디바이스)

### 칼럼 관리

**FR-009: 칼럼 표시** (Must-have)

- **Description:** 3개의 고정된 칼럼이 표시되어야 함
- **Acceptance Criteria:**
  - [ ] "할 일", "진행 중", "완료" 3개 칼럼이 순서대로 표시
  - [ ] 각 칼럼은 명확한 제목과 구분선을 가져야 함
  - [ ] 각 칼럼 헤더에 현재 카드 개수 표시
  - [ ] 빈 칼럼에는 "카드 없음" 또는 안내 메시지 표시

**FR-010: 칼럼별 카드 정렬** (Should-have)

- **Description:** 각 칼럼 내 카드의 순서를 관리할 수 있어야 함
- **Acceptance Criteria:**
  - [ ] 기본적으로 최신 카드가 상단에 표시
  - [ ] 드래그로 카드 순서를 수동 조정 가능
  - [ ] 순서 변경 후 위치가 유지되어야 함

### 사용자 인터페이스

**FR-011: 반응형 레이아웃** (Must-have)

- **Description:** 다양한 화면 크기에 맞게 레이아웃이 조정되어야 함
- **Acceptance Criteria:**
  - [ ] 데스크톱(1024px+): 3개 칼럼이 가로로 나란히 표시
  - [ ] 태블릿(768px-1023px): 2개 칼럼 표시, 스크롤 가능
  - [ ] 모바일(<768px): 1개 칼럼 표시, 탭으로 칼럼 전환 또는 세로 스크롤
  - [ ] 모든 해상도에서 가독성 유지

**FR-012: 시각적 피드백** (Must-have)

- **Description:** 사용자 액션에 대한 즉각적인 시각적 피드백 제공
- **Acceptance Criteria:**
  - [ ] 버튼 호버/클릭 시 시각적 변화
  - [ ] 카드 생성/수정/삭제 시 성공/실패 메시지 표시
  - [ ] 드래그 앤 드롭 중 상태 표시
  - [ ] 로딩 상태 표시 (데이터 저장/불러오기 시)

**FR-013: 키보드 접근성** (Should-have)

- **Description:** 키보드만으로 모든 기능을 사용할 수 있어야 함
- **Acceptance Criteria:**
  - [ ] Tab 키로 모든 인터랙티브 요소 접근 가능
  - [ ] Enter/Space로 버튼 활성화
  - [ ] ESC로 모달/편집 모드 닫기
  - [ ] 포커스가 시각적으로 명확히 표시
  - [ ] 화살표 키로 카드 간 이동 (Nice-to-have)

### 데이터 관리

**FR-014: 데이터 저장** (Must-have)

- **Description:** 카드 데이터가 영구적으로 저장되어야 함
- **Acceptance Criteria:**
  - [ ] 카드 생성/수정/삭제/이동 시 즉시 저장
  - [ ] 페이지 새로고침 후에도 데이터 유지
  - [ ] 브라우저 종료 후 재접속 시 데이터 유지
  - [ ] 저장 실패 시 사용자에게 알림

**FR-015: 데이터 불러오기** (Must-have)

- **Description:** 앱 시작 시 저장된 카드 데이터를 불러와야 함
- **Acceptance Criteria:**
  - [ ] 페이지 로드 시 자동으로 데이터 불러오기
  - [ ] 로딩 중 로딩 인디케이터 표시
  - [ ] 데이터가 없을 경우 빈 보드 표시
  - [ ] 불러오기 실패 시 에러 메시지 및 재시도 옵션

## Non-Functional Requirements

### Performance

**NFR-001: 드래그 앤 드롭 성능**

- 드래그 중 60fps 유지 (16.67ms 이하의 프레임 시간)
- 카드 이동 시 지연 시간 50ms 이하
- 100개 카드까지 성능 저하 없이 처리

**NFR-002: 페이지 로딩**

- 초기 페이지 로드 시간 2초 이내
- Time to Interactive (TTI) 3초 이내
- 데이터 불러오기 1초 이내

**NFR-003: 애니메이션**

- 모든 전환 애니메이션은 부드럽게 (ease-in-out)
- 카드 이동 애니메이션 200-300ms
- 상태 변경 시 즉각적인 피드백 (50ms 이내)

### Security

**NFR-004: 데이터 보호**

- 사용자 데이터는 안전하게 저장되어야 함
- XSS 공격 방지 (사용자 입력 sanitization)
- CSRF 보호 (API 통신 시)

**NFR-005: 입력 검증**

- 모든 사용자 입력은 클라이언트 및 서버에서 검증
- SQL Injection 방지 (해당 시)
- 파일 크기 및 타입 제한 (첨부 파일 기능 시)

### Accessibility

**NFR-006: WCAG 2.1 준수**

## Constraints & Assumptions

### Technical Constraints

**TC-001: 프론트엔드 중심**

- 초기 버전은 프론트엔드 단일 페이지 애플리케이션(SPA)으로 구현
- 백엔드 없이 로컬 스토리지 또는 IndexedDB 사용

**TC-002: 웹 표준 기술**

- 최신 웹 표준 (HTML5, CSS3, ES6+) 사용
- 모던 브라우저 API 활용 (Drag and Drop API, Local Storage API)

**TC-003: 오프라인 우선 아님**

- 초기 버전은 온라인 상태 가정
- 오프라인 지원은 추후 고려 사항

### Business Constraints

**BC-001: 개인 프로젝트**

- 단일 사용자용 애플리케이션
- 다중 사용자/협업 기능은 스코프 외

**BC-002: 빠른 MVP 개발**

- 핵심 기능(카드 CRUD + 드래그 앤 드롭) 우선 구현
- 부가 기능은 이후 단계에서 추가

**BC-003: 비용 제약**

- 무료 또는 저비용 호스팅 솔루션 사용
- 외부 유료 서비스 최소화

### Assumptions

**AS-001: 사용자 환경**

- 사용자는 모던 브라우저를 사용한다고 가정
- 사용자는 마우스 또는 터치스크린을 사용 가능하다고 가정
- 사용자는 인터넷에 연결되어 있다고 가정 (초기 버전)

**AS-002: 데이터 규모**

- 한 사용자당 최대 100개 카드를 관리한다고 가정
- 카드 제목 평균 50자, 설명 평균 200자로 가정

**AS-003: 사용 패턴**

- 사용자는 하루에 10-20회 카드 상태를 변경한다고 가정
- 주로 데스크톱에서 사용하며, 모바일은 보조적 사용

**AS-004: 보안**

- 초기 버전은 민감한 정보를 다루지 않는다고 가정
- 인증/인가 기능은 추후 추가
